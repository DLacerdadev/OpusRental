TAREFA 1 — Criar whatsapp.service.ts com mock mode
=======================================================

Crie server/services/whatsapp.service.ts com a seguinte
interface e comportamento:

INTERFACE PÚBLICA (não mude — T5 vai consumir essa interface):

  interface WhatsAppMessage {
    to: string;           // número com código do país: "15551234567"
    body: string;         // texto da mensagem
    metadata?: {
      eventType: string;  // "payment_generated", "invoice_issued", etc.
      entityId?: string;
      tenantId?: string;
    };
  }

  interface WhatsAppResult {
    success: boolean;
    messageId?: string;
    error?: string;
    mode: "mock" | "live";
  }

  class WhatsAppService {
    static async sendMessage(msg: WhatsAppMessage): Promise<WhatsAppResult>
    static isConfigured(): boolean
    static getProvider(): string
  }

COMPORTAMENTO:
  - Se WHATSAPP_API_KEY não estiver definida no ambiente:
    → mode = "mock"
    → Loga no console com formato padronizado (veja abaixo)
    → Retorna { success: true, mode: "mock" }
  - Se WHATSAPP_API_KEY estiver definida:
    → Usa WHATSAPP_PROVIDER para escolher o adaptador
    → Chama a API real
    → Retorna { success: true, messageId: "...", mode: "live" }

LOG PADRÃO DO MOCK (use exatamente esse formato):
  [WhatsApp MOCK] {timestamp UTC}
    Provider : mock
    To       : {número}
    Event    : {eventType}
    Message  : {primeiros 80 chars do body}...

TAREFA 3 — Implementar adaptadores de provedor
=======================================================

Dentro de whatsapp.service.ts, implemente os adaptadores
para os dois provedores suportados:

PROVEDOR 1: Twilio (WHATSAPP_PROVIDER=twilio)
  Requer variáveis:
    TWILIO_ACCOUNT_SID
    TWILIO_AUTH_TOKEN
    TWILIO_WHATSAPP_FROM  (ex: "whatsapp:+14155238886")

  Endpoint:
    POST https://api.twilio.com/2010-04-01/Accounts/{SID}/Messages.json
    Auth: Basic (AccountSid:AuthToken)
    Body (form-urlencoded):
      From: {TWILIO_WHATSAPP_FROM}
      To:   whatsapp:+{msg.to}
      Body: {msg.body}

  Retorno esperado: { sid: "SM..." } → use como messageId

PROVEDOR 2: Z-API (WHATSAPP_PROVIDER=zapi)
  Requer variáveis:
    ZAPI_INSTANCE_ID
    ZAPI_TOKEN
    ZAPI_CLIENT_TOKEN

  Endpoint:
    POST https://api.z-api.io/instances/{INSTANCE_ID}/token/{TOKEN}/send-text
    Header: Client-Token: {ZAPI_CLIENT_TOKEN}
    Body (JSON):
      { "phone": "{msg.to}", "message": "{msg.body}" }

  Retorno esperado: { zaapId: "..." } → use como messageId

FALLBACK: Se WHATSAPP_PROVIDER não for reconhecido,
  logar erro e retornar { success: false, error: "Provider not supported", mode: "live" }

VALIDAÇÃO — Como verificar que funcionou
=======================================================

1. O servidor deve iniciar sem erros após suas mudanças.

2. Em modo MOCK (sem WHATSAPP_API_KEY), ao chamar
   o endpoint de teste, o console deve exibir:
   [WhatsApp MOCK] ...

3. O endpoint POST /api/whatsapp/test deve:
   - Retornar 401 se não autenticado
   - Retornar 403 se não for admin
   - Retornar 200 com { success: true, mode: "mock" }
     quando autenticado como admin sem credenciais reais
   - Retornar 400 se "to" for inválido

4. WhatsAppService.isConfigured() deve retornar:
   - false quando WHATSAPP_API_KEY não está definida
   - true quando está definida

5. Nenhum arquivo de servidor existente deve ter
   sua lógica alterada — apenas adições.
